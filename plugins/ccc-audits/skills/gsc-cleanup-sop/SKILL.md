# GSC Cleanup SOP v2.0

**Workflow: Context → Analyze → Diagnose → Fix → Document.**
You drive Chrome directly through all four phases. The user provides access and context; you do the work.

---

## Phase 0: Context Intake

**Start here — before touching Chrome.** Your first message should confirm you'll be driving the browser and ask for what's missing.

Open with something like: *"I'll drive Chrome directly to run the GSC cleanup. Is GSC already open in a Chrome tab? If not, I'll navigate to https://search.google.com/search-console now. Also, could you share: (1) the list of key content pages that absolutely need to rank, and (2) which plugins are running on the site? Particularly: is WooCommerce active? Any multilingual plugin (WPML, Polylang)? Which theme?"*

**Required before starting Phase 1:**
- Client name and domain (e.g., `example.de`)
- Confirm Chrome/GSC access: Is GSC already open? If not, navigate there.
- List of key content pages (the "money pages" that should rank)

**Required for diagnosis — ask explicitly:**
- **Which plugins are active?** Particularly: WooCommerce, WPML, Polylang, Elementor, FacetWP, SearchWP, or any filter/shop plugins
- **Which theme?** (Avia/Enfold, Divi, Elementor, Astra, GeneratePress, custom, etc.)
- If WooCommerce is present: Is there an active shop, or was it installed by accident?

**Why the plugin/theme question matters:** The root cause of crawl budget waste is almost always a specific plugin generating parameter URLs. WooCommerce, multilingual plugins, faceted search plugins, and some themes all produce different URL patterns. You need to know which one is installed to diagnose correctly and choose the right robots.txt template. A site without WooCommerce cannot have WooCommerce crawl budget waste — so never assume.

## Phase 1: GSC Browser Analysis

Get the tab context and take a screenshot to confirm the right GSC property is active (check domain in URL).

### 1a. Coverage Overview

Navigate to: **Indexing → Pages** (left sidebar)

Read the breakdown of non-indexed pages by reason. The count next to each reason tells you almost everything you need to know about root cause:

**Critical diagnosis rule — apply before anything else:**

> **If "Gecrawlt – zurzeit nicht indexiert" (Crawled – currently not indexed) is over 100 pages on a site that has fewer than ~200 real content pages → the primary root cause is almost certainly parameter URL explosion consuming crawl budget.**
>
> This is the single most common pattern on WordPress sites. Google wastes its daily crawl budget on hundreds of URL variations generated by plugins or the theme, and never gets to the actual content pages. **Lead with this diagnosis.** Do not lead with Core Web Vitals, canonical confusion, or JavaScript rendering as the primary cause — those are secondary possibilities.
>
> **Identify which plugin is the culprit based on what's installed:**
> - **WooCommerce** → product filter variations (`?pa_color=red&pa_size=large`), cart/checkout/account pages, `/?wc-ajax=`
> - **Avia/Enfold theme** (on top of WooCommerce) → `/?avia_extended_shop_select=VALUE`, `/?avia=`
> - **WPML or Polylang** → language parameter variations (`?lang=de`, `/de/`, `/en/`), duplicate content in multiple languages
> - **FacetWP or Search & Filter plugins** → faceted search URLs (`?fwp_...=`, `?filter_...=`)
> - **Elementor** (rare but possible) → builder-specific URL parameters in preview/draft states
> - **Unknown** → note the GSC "Gecrawlt" URLs to identify the pattern visually
>
> If WooCommerce is installed but there's no real shop (common — installed by accident), removing WooCommerce entirely would be the single most impactful SEO fix. Don't recommend this for clients who actually use their shop.

Other signals to note:
- **"Nicht gefunden (404)"**: Broken internal links or recently deleted content
- **"Durch 'noindex'-Tag ausgeschlossen"**: Usually intentional — verify before flagging. Common legitimately-noindexed pages: wp-login.php, /feed/, cart, checkout, my-account, search results, language-switcher pages (WPML)
- **"Duplikat"**: Canonical issues, often secondary to the crawl budget problem

### 1b. Sitemap Audit

Navigate to: **Indexing → Sitemaps**

Check each submitted sitemap:
- ✅ Status "Erfolgreich" + XML extension = correct
- 🚨 **"Sitemap ist HTML"** = someone submitted a content page URL instead of an XML file — must delete (see Phase 2)
- 🚨 **"Konnte nicht abgerufen werden"** = broken sitemap URL

Note the "Entdeckte Seiten" count per sitemap.

### 1c. noindex Validation Check

If you see **"Durch 'noindex'-Tag ausgeschlossen" with "Validierung fehlgeschlagen"** status, click that row and extract the URLs:

```javascript
const rows = document.querySelectorAll('table tr');
const urls = [];
rows.forEach(tr => {
  const cells = tr.querySelectorAll('td');
  if (cells.length > 0) {
    const urlText = cells[0].textContent.trim();
    if (urlText.startsWith('http')) urls.push(urlText);
  }
});
JSON.stringify(urls);
```

Check each URL against platform-specific known-safe noindex patterns:
- **WordPress (all):** `wp-login.php`, `/feed/`, `/wp-admin/`
- **WooCommerce:** `/cart/`, `/checkout/`, `/my-account/`, product feed URLs
- **WPML/Polylang:** language-switcher redirect pages, duplicate translated pages
- **Search/filters:** search result pages, faceted filter result pages

Actual content pages (service pages, blog posts, landing pages) with noindex → investigate immediately.

"Validierung fehlgeschlagen" on intentional noindex pages is usually a false alarm from someone clicking "Validate Fix" by mistake.

### 1d. Key Content Page Status

For each page in the user's key content list, use URL Inspection.

**Reliable navigation method (React SPA):**
1. Click the search bar at top
2. Triple-click to select all existing text
3. Type the full URL (e.g., `https://domain.de/my-page/`)
4. Press Enter

Note for each page: indexed or not, last crawl date, referring pages.

## Phase 3: Quick Fixes (Sofortfixes)

These can be done directly in Chrome right now.

### Fix 1: Delete Wrong Sitemaps

For sitemaps with "Sitemap ist HTML" status:

1. Navigate to **Indexing → Sitemaps**
2. Find the row and navigate to its detail page via JavaScript:
```javascript
const rows = document.querySelectorAll('tr');
rows.forEach(tr => {
  if (tr.textContent.includes('TARGET_URL_FRAGMENT')) {
    tr.click();
  }
});
```
3. On the detail page, find the ⋮ "Weitere Optionen" button and select "Sitemap entfernen"
4. Screenshot to confirm "Sitemap entfernt"

**Note:** The delete option is only available on the detail page, not the overview.

### Fix 2: Submit Re-indexing Requests

For key content pages that showed "URL ist nicht auf Google":

1. Navigate to **URL-Prüfung** in sidebar
2. For each page: triple-click search bar → type URL → Enter → wait for load
3. If "URL ist nicht auf Google" → click "INDEXIERUNG BEANTRAGEN"
4. Confirm "Indexierung wurde beantragt" dialog → click "Schließen"
5. Repeat (GSC limit: ~10 requests/day — prioritize money pages first)

## Phase 4: WordPress Recommendations

Document these for the client/developer — they require WordPress access.

### robots.txt

**IMPORTANT: Do not generate robots.txt rules from scratch. Open `references/robots-txt-templates.md` and select the appropriate template based on the client's plugin stack.** Adapt only the domain name in the Sitemap directive. Using the tested templates ensures all critical parameter patterns are covered — a self-generated template will miss plugin-specific patterns.

**Template selection logic:**

| Client setup | Template to use |
|---|---|
| WordPress only (no shop, no special plugins) | WordPress Baseline |
| WordPress + WooCommerce, no specific theme | WooCommerce |
| WordPress + WooCommerce + Avia/Enfold theme | WooCommerce + Avia/Enfold |
| WordPress + WPML or Polylang | WPML / Polylang |
| WordPress + FacetWP or Search & Filter | Faceted Search |
| Multiple plugins active | Combine relevant blocks from multiple templates |

After selecting the template, compare it to the client's actual `https://[domain]/robots.txt` and note specifically what's missing.

### Plugin Assessment

After diagnosing the crawl budget waste, ask the right follow-up question based on the culprit:

- **WooCommerce installed but no active shop:** Recommend deinstalling WooCommerce entirely — eliminates hundreds of phantom parameter URLs overnight. Most impactful single fix.
- **WooCommerce with active shop:** Recommend Yoast WooCommerce SEO for canonical tags (`/shop/?*` → `/shop/`)
- **WPML/Polylang:** Check if all translated pages are intentional; ensure hreflang tags are properly set
- **FacetWP/Search & Filter:** Ensure filter result pages are excluded from sitemap and blocked in robots.txt

### Other fixes to document:
- Fix sitemap HTTP→HTTPS if needed
- Add 301 redirects for any 404 errors found
- noindex empty category/tag pages

## Rules (Update When Things Go Wrong)

1. **Never use the `wait` tool** — causes Chrome extension disconnects. Poll with repeated screenshots.
2. **GSC URL navigation (React SPA quirk)**: Triple-click the search bar → type URL → press Enter. Never use JS to set the input value and dispatch key events — this triggers indexing of the current page instead of navigating.
3. **Sitemap delete requires detail page**: The ⋮ delete option is only on the sitemap's detail page. Use `tr.click()` via JS to navigate there since direct URL navigation returns 404.
4. **"Validierung fehlgeschlagen" on noindex is usually a false alarm**: Always verify the actual URLs before flagging as a problem.
5. **Always close the indexing success dialog** before navigating to the next URL.
6. **"Gecrawlt – zurzeit nicht indexiert" at high counts = parameter URL explosion** — not Core Web Vitals, not JavaScript rendering, not canonical confusion. Identify which plugin based on what's installed. Lead with this diagnosis.
7. **robots.txt: use the templates, don't generate your own** — see `references/robots-txt-templates.md`. Select the template matching the client's plugin stack. Self-generated templates miss plugin-specific patterns. Multiple templates can be combined when multiple plugins are active.

## Self-Improvement

When a new GSC quirk is discovered, add it to Rules above. When a diagnosis pattern repeats across multiple clients, consider adding it as an explicit shortcut in Phase 1.
